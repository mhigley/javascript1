Function Hoisting

Hoisting was thought up as a general way of thinking about how execution context (specifically the creation and execution phases) work in JavaScript. But, hoisting can lead to misunderstandings. Hoisting teaches that variable and function declarations are physically moved to the top of your coding, but this is not what happens at all. What does happen is that variable and function declarations are put into memory during the compile phase, but stays exactly where you typed it in your coding.

For an example using functions, let's console the model of our vehicle:

function myCar(vehicle ) {
console.log (vehicle); // wrangler
}
myCar('wrangler') ;

No surprise that this works. But what about this?

myCar('wrangler') ;
function myCar(vehicle ){
console.log (vehicle); // wrangler
}

It still works because of how context excecution works in JavaScript. When the script is compiled, the function declaration myCar is placed into memory. But what about function expressions you may ask?

myCar('wrangler') ;
var myCar = function (vehicle) {
console.log (vehicle); // TypeError: myCar is not a function
};

Notice that the function expression is nothing more than an anonymous function declared as a variable. Variable names and function declarations get hoisted!
